<h2 id="binding">Payer &amp; Amount Binding</h2>

A proof binds to the exact <strong>merchant</strong>, <strong>payer</strong>, and <strong>amount</strong> it was generated for. The contract re-derives each value from the real transaction arguments and rejects the payment if any of them don't match — a captured proof cannot be replayed with a different recipient, a different wallet, or a different amount.

<ThreeCol items={[
  ["Recipient binding", "recipientHash = hash(merchant). Prevents a proof from being redirected to a different merchant."],
  ["Payer binding", "payerHash = hash(sender). Prevents a broadcast-but-unconfirmed proof from being resubmitted by a different wallet, which would otherwise burn the original sender's nullifier at no cost to the attacker."],
  ["Amount binding", "amount is a public signal, bound exactly. The circuit performs no range checks — any spending-policy limits belong in the application, not the circuit."],
]} />

<h2 id="root-operator">Root Operator &amp; Automated Sync</h2>

The Merkle root is anchored on-chain by an automated job, not a manually-invoked admin command. The contract separates two roles:

<RoleCards />

A GitHub Actions job calls the sync endpoint every ~10 minutes. It compares the on-chain root against the current commitment store, and submits a new root — signed by the RootOperator key — only when they differ. The endpoint is protected by a shared secret, a distributed lock (so overlapping runs don't race each other), and an idempotency check (a free on-chain read before every write) so a no-op cycle costs nothing.

<h2 id="root-history">Root History Window</h2>

The contract keeps the last <strong>30 anchored roots</strong>, not just the latest one. A proof is accepted if its root matches any entry in that window. This closes a race that would otherwise occur with frequent syncing: a user fetches a proof against root N, the sync job anchors root N+1 while they're still generating the proof, and the payment would previously fail outright with a hard <code>RootMismatch</code> error. Now it still succeeds, as long as the sync job hasn't cycled past 30 roots in the meantime.

<h2 id="nullifier">Nullifier Replay Protection</h2>

Each ZK payment proof includes a <strong>nullifier</strong>: <code>Poseidon(NULLIFIER_TAG=1, secret, nonce)</code> — a random one-time value derived from the secret and a CSPRNG nonce.
